# 叙事与人生曲线 · 统一规划

---

## 一、不同身份的人生曲线

### 1.1 目标

在**时间聚合（Collection / Moment Ultra）页内新增「人生曲线」区块，展示选定时间范围内各身份（role）的占比**与**重点事件**，形成身份平衡的可视化与叙事锚点。

### 1.2 呈现形式

- **各身份占事件总数比例**
  - 指标：以 **total number of events**（总事件数）为分母，各 role 的事件数为分子，得到各身份占比。
  - 展示：一条**堆叠条**（100% 宽度），按各 role 占比分段，每段用 [getRoleColor](src/lib/constants/roles.ts) 上色；左侧或下方**图例**列出角色名 + 占比百分比（如「创作者 32%」「执行者 45%」）。
  - 时间范围：用户可选「本周 / 本月 / 近 3 个月」，控制聚合区间。
  - 可选：增加「按事件数 / 按时长」切换，按时长时分母为总时长、分子为各 role 时长。
- **趋势线（进阶视觉）**
  - 产品价值：「变化趋势」比「静态占比」更能促使用户反思「我最近变了吗？」。
  - 实现建议：在 Collection 页人生曲线区块内，**在堆叠条上方**增加一条小的**趋势线**：对比当前周期与上一周期（如本周 vs 上周、本月 vs 上月）各 role 占比变化。
  - 展示示例：图例或趋势区显示「学者 +10%」「执行者 −5%」等，让用户一眼看到各身份占比的升降；实现时可取两段区间的聚合结果做差或百分比差，用简约的 mini 条或文字标签呈现。
- **重点事件：Starred（待办）与 Highlight（高光）区分**
  - 产品定义：**Starred** 用于任务管理（GTD/待办）；**Highlight** 用于 Life Books 共创（叙事/生命高光）。用户可在新增/编辑日程时分别勾选「星标」与「高光」。
  - 数据：事件模型增加 `**starred?: boolean`**、`**highlight?: boolean`**。[ScheduleEvent](src/types.ts) 与 [eventsRepo](src/lib/repositories/eventsRepo.ts)、Supabase `events` 表增加两列（如 `starred boolean default false`、`highlight boolean default false`）。
  - 新增/编辑入口：在 [AddEventModal](src/components/AddEventModal.tsx) 中增加两个选项——「星标」（★ 图标，GTD 重点）、「高光」（✨ 图标，叙事里程碑），保存时分别写入 `event.starred`、`event.highlight`。
  - 人生曲线区块内：在堆叠条与图例下方展示**本周期重点事件**列表时，可分区或标签区分「星标」与「高光」；**周报/Chapter 生成时，AI 优先将 `highlight === true` 的事件视为「本周里程碑」**，降低总结噪点、突出叙事锚点。

### 1.3 放置位置

- [CollectionView](src/components/CollectionView.tsx) 左侧列（与 [StatsSummaryView](src/components/StatsSummaryView.tsx) 同列），放在统计卡片**下方**。人生曲线卡片**右上角预留「一键加入我的 Life Book」**入口（见第五部分），便于后续共创无缝接入。
- 可选将「人生曲线」封装为 [RoleBalanceCard.tsx](src/components/RoleBalanceCard.tsx)，接收 `events`、`completedInstances`、`language`、`getRoleColor`、`getRoleDisplayName`。

### 1.4 数据与依赖

- 数据：用 [expandRecurringEvents](src/lib/events.ts) 在选定范围内展开事件，按 `event.role` 分组统计 count（及时长）；总事件数 = 展开后实例总数。
- 重点事件：需新增 `starred`、`highlight` 两字段并持久化；Timeline/列表展示时星标与高光可用不同图标（★ / ✨）。

## 二、叙事与身份功能（7 项）

### 2.1 主题线：用现有 tag 聚合成「主题时间线」视图

- **产品定义**：按 tag 看事件流——用户选定一个或多个 tag，查看带有该 tag 的事件在时间轴上的分布。
- **放在哪里**：时间聚合页内 Tab 或区块「主题线」。左侧保留 Stats + 人生曲线；右侧或下方为「主题时间线」：选 tag → 展示该 tag 下事件按日/周的时间流（列表或简版日历高亮）。区块右上角预留「加入我的 Life Book」入口（见第五部分）。
- **实现要点**：复用 `events` + `expandRecurringEvents`，按 `event.tags` 过滤；Tag 选择器 + 时间范围 + 事件列表按日期分组。无新表。

---

### 2.2 角色能量条（Role Energy Bar）与平衡提醒

- **产品定义**：在 Stats & Summary 中按角色统计事件数/时长占比；当某角色占比过高（如 ≥80%）时，友善提醒建议平衡其他角色（如「静修者」）。
- **放在哪里**：[StatsSummaryView](src/components/StatsSummaryView.tsx) 所在卡片内，在现有指标之下增加「角色能量条」+「平衡提醒」文案。
- **实现要点**：与人生曲线共用按 role 聚合逻辑；堆叠条 + 图例；条件渲染提醒文案（阈值 75–80%，中英模板）。

---

### 2.3 Chapter 叙事摘要 + 定期推送 + 用户可编辑

- **产品定义**：基于章节 + 事件 + 日记的叙事摘要生成（周/月），应用内提示「Chapter N 可查看」，用户可编辑并沉淀。
- **放在哪里**：Collection 右侧「共创区」改为章节叙事入口：展示最近章节、按钮「生成本章节」/「查看 Chapter 1/2」；点击进入可编辑（章节名、叙事摘要、反思题）。定期推送 = 应用内根据「上次生成时间」与当前日期展示 banner 提示。
- **实现要点**：调用已有 [generateWeeklyChapter](src/lib/gemini.ts)；章节结果存 localStorage 或 Supabase `chapters` 表；UI 支持 3 选 1 标题 + 编辑摘要 + 保存。

---

### 2.4 叙事闭环：当日输入 → 周回顾 → 月总结 → 下月目标

- **产品定义**：周回顾（模板 + AI 预填）→ 月总结（模板 + AI）→ 用户确认/改写 → 下月重点写入设置或 day_meta，形成闭环。
- **放在哪里**：Collection 或日视图入口「本周回顾」「本月总结」；月总结最后一步「下月重点」存 `AppSettings` 扩展或 `day_meta`。
- **实现要点**：先定义模板与触发时机（周回顾：周一/周日；月总结：月末/月初）；周回顾复用 `generateWeeklyChapter`；月总结新增 `generateMonthlyChapter` 或月维度 prompt；下月目标展示在日视图/月视图顶部。

---

### 2.5 每日总结改为「意义总结」+ 2.6 事件级意义微标注（与 AI 总结整合）

- **产品定义**：每日总结强调**意义**（叙事密度）；事件级可写一句「意义」，汇总成意义总结。**事件意义 (meaning) 作为 AI 的输入源**，避免「每天写意义」与「AI 生成总结」在用户心智中重复。
- **整合逻辑**：用户输入的「意义」= **叙事锚点**；AI 生成的「AI 总结」= **叙事润色**。当用户填写了 `meaning`，[generateWeeklyChapter](src/lib/gemini.ts)（及日总结/月总结）应**优先引用这些 meaning 作为素材**，使 AI 输出明确体现用户自述，用户感到「我的输入被尊重」而非仅为填表。
- **放在哪里**：[DailyJournal](src/components/DailyJournal.tsx) 标题/placeholder 改为意义向；[AddEventModal](src/components/AddEventModal.tsx) 增加「意义（一句）」输入。
- **实现要点**：
  - 意义总结：[generateDailySummary](src/lib/gemini.ts) prompt 改为基于事件 + **事件意义句 (meaning)** 生成「今日意义」；DailyJournal snippet 增加意义向句式。
  - 事件意义：方案 A 约定 `description` 首行为意义；方案 B 新增 `ScheduleEvent.meaning?: string` 及 DB 列。AddEventModal 增加一句意义输入与引导文案（如「给这件事写一句意义，方便日后回顾与 AI 总结」）。
  - AI 输入约定：所有周/月章节与按 tag/role 总结的 prompt 中，传入事件时附带 `meaning` 字段，并明确 instruct「优先引用用户填写的 meaning 作为叙事锚点，再在此基础上润色与串联」。

---

### 2.7 善用 AI 根据 event tag / role 总结

- **产品定义**：在 Collection 中按时间范围 + 维度（tag 或 role）用 AI 生成 2–4 句总结。生成时**优先引用事件上的 meaning** 作为素材，与 2.5/2.6 形成「用户意义 → AI 润色」的闭环。
- **放在哪里**：Collection 内「AI 总结」卡片：选时间范围 + 选 tag/role → 点击生成 → 展示文案（可复制/写入章节）。
- **实现要点**：新增 `generateSummaryByFilter(events, language, { byTag?, byRole? })`，传入的 events 需包含 `meaning` 字段；prompt 中要求优先引用用户填写的 meaning；UI 下拉选维度与具体 tag/role，展示结果。

## 三、落点总览与推荐顺序

**推荐顺序微调**：将 **2.6 事件级意义微标注** 提至 **第一批** 实现。理由：这是叙事的**源头数据**；若不在添加日程时就记录 Tag + Role + Meaning，后续 AI 总结与人生曲线会缺乏深度。让用户在新增日程时习惯性写「一句意义」，是养成叙事习惯的关键动作。


| 功能                                | 主要落点                                | 推荐顺序         |
| --------------------------------- | ----------------------------------- | ------------ |
| **2.6 事件级意义微标注**                  | AddEventModal + `meaning` 字段        | **1（优先）**    |
| 人生曲线（比例 + 趋势 + Starred/Highlight） | Collection 左侧，Stats 下方              | 1            |
| 主题线                               | Collection Tab/区块「主题线」              | 2            |
| 角色能量条 + 平衡提醒                      | StatsSummaryView 内                  | 2（与人生曲线共用聚合） |
| 意义总结（2.5）                         | DailyJournal + generateDailySummary | 3            |
| Chapter 叙事摘要 + 推送                 | Collection 右侧共创区                    | 4            |
| 叙事闭环                              | Collection 周回顾/月总结入口 + 下月目标         | 5            |
| AI 按 tag/role 总结                  | Collection「AI 总结」卡片                 | 6            |


**建议实现批次**

- **第一批**：**事件级意义（2.6）**——`meaning` 字段、AddEventModal 意义输入与引导；**人生曲线（1）**——含 `starred`/`highlight`、AddEventModal 星标/高光、RoleBalanceCard 比例条 + 趋势线 + 重点/高光事件列表。
- **第二批**：角色能量条与提醒（2.2）+ 主题线（2.1）——Stats 增强 + Collection Tab。
- **第三批**：意义总结（2.5）——DailyJournal 与 generateDailySummary 以 meaning 为锚点；AI 总结（2.7）与 generateWeeklyChapter 优先引用 meaning、highlight。
- **第四批**：Chapter 叙事与推送（2.3）+ 叙事闭环（2.4）。

## 四、AI 总结需考虑的因素（输入清单）

所有「AI 总结」类能力（日总结、周章节、月总结、按 tag/role 总结）在构造 prompt 时，应尽量纳入以下因素，使输出有叙事密度且尊重用户输入。


| 因素                              | 数据来源                                                      | 说明与用法                                                                                                    |
| ------------------------------- | --------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **Day name（当日/当周某日的「章节名」）**     | `day_meta.day_name` + `day_name_is_manual`                | 用户为某天起的名字（如「咖啡与截止日」）。**若 `day_name_is_manual === true`，表示用户手动修改过，应视为强信号**：总结中可引用或呼应该名字，体现「用户给这一天定的调性」。   |
| **Day tag**                     | `day_meta.day_tag`（或前端 `dayTags[date]`）                   | 当日主题/类型标签（如「深度工作」「休息日」）。用于按日聚合时的主题线索，周/月总结中可概括「本周/月主要标签分布」。                                              |
| **Event role（身份标签）**            | `ScheduleEvent.role`                                      | 事件所属身份（创作者、学者、静修者等）。周/月总结与按 role 总结时**按角色聚类**，突出身份平衡或主旋律。                                                |
| **Event tags / label**          | `ScheduleEvent.tags`、`ScheduleEvent.label`                | 事件级标签与自定义 label。按 tag 总结时过滤与分组；日/周总结中可提及「标签维度」的分布或重点。                                                    |
| **Energy / Mood / Focus（当日状态）** | `day_meta.energy_level`、`mood_level`、`focus_level`（0–100） | 用户填写的当日能量、情绪、专注度。**日总结**可据此写「今日状态」一句；周/月总结可概括趋势（如「本周后半能量回升」），不虚构具体数字。                                    |
| **重要事件特殊标记**                    | `ScheduleEvent.starred`、`ScheduleEvent.highlight`         | **Starred**：GTD/待办重点，总结中可点出「重点完成情况」。**Highlight**：叙事高光/里程碑，**周章节与月总结中优先将 highlight 事件视为锚点**，减少流水账、突出故事线。 |
| **事件意义（一句）**                    | `ScheduleEvent.meaning`                                   | 用户为事件写的一句意义。**所有总结类 prompt 均应优先引用用户填写的 meaning 作为叙事锚点**，再在此基础上润色与串联，体现「用户输入被尊重」。                         |
| **日记 / Journal**                | `day_meta.journal`（或 `journalEntries[date]`）              | 当日/当周用户手写或语音的日记。日总结可与之融合；周章节已在使用，需与 meaning、highlight、day name 等一起作为叙事素材。                                |
| **完成情况**                        | `ScheduleEvent.completed`                                 | 事件是否完成。总结中可简要体现「完成率」「重点完成项」，避免只列计划不列结果。                                                                  |


**实现约定**

- **日总结**（`generateDailySummary`）：入参建议扩展为「当日 events + 当日 day name（及 isManual）+ 当日 day tag + 当日 energy/mood/focus + 当日 journal」；事件列表传入时附带 `meaning`、`starred`、`highlight`、`role`，prompt 中要求优先引用 meaning、突出 highlight/starred。
- **周章节 / 月总结**（`generateWeeklyChapter`、`generateMonthlyChapter`）：除事件列表（含 meaning、role、starred、highlight）与 journal 外，增加「当周/月每日 day name（标注是否手动）」「day tag 分布」「可选：能量/情绪趋势摘要」；prompt 中明确 highlight 为里程碑、meaning 为叙事锚点。
- **按 tag/role 总结**（`generateSummaryByFilter`）：传入已过滤的 events（含 meaning、highlight、starred、role），prompt 要求优先引用 meaning、突出 highlight，并按所选维度归纳。

## 五、周章节叙事 AI（人生档案整理师）参考

产品底层逻辑已写在 [feather-schedule-unified.plan.md](.cursor/plans/feather-schedule-unified.plan.md) 第五部分；实现入口为 [generateWeeklyChapter](src/lib/gemini.ts)。本章节叙事输出（3 个章节名、叙事摘要、2 个反思题）可供「Chapter 叙事摘要」「周回顾」复用。

- **Highlight 权重**：调用时在输入中标注 `highlight === true` 的事件，供 AI 视为「本周里程碑」优先写入叙事，降低噪点。
- **Meaning 锚点**：传入事件时附带用户填写的 `meaning`，prompt 中明确「优先引用用户填写的 meaning 作为叙事锚点，再润色与串联」，使用户感到输入被尊重。

## 六、Life Books 共创：导出/沉淀思维

为在「Life Books 共创」阶段不显臃肿、且无缝接入，建议在 **UI 设计上加入“导出/沉淀”思维**：

- **预留入口**：所有**聚合视图**（如主题线、人生曲线、角色能量条所在区块、Chapter 列表等）在**右上角**预留 **「一键加入我的 Life Book」** 按钮（或等价文案，如「装订到 Life Book」）。
- **产品暗示**：无论当前是否已开通共创功能，该按钮都暗示：「你当下的生活痕迹，是有被装订成册的价值的。」便于后续真正接入 Life Book 时，仅需打通该入口的逻辑与权限，而不必再改版布局。
- **涉及范围**：Collection 页内人生曲线卡片、主题线区块、Chapter/周回顾卡片等聚合视图的卡片头部或右上角，统一预留该按钮位；样式可先为次要按钮或文字链，避免喧宾夺主。
